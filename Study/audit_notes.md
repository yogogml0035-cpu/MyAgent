# Study 资料审计记录

本文件记录这套学习资料的维护原则，帮助后续修改时保持“可学习、可验证、可自测”。

## 审计结论

本轮修补后的资料已经形成了比较稳定的三层结构：

1. 入口层：`README.md`、`learning_plan.md`、`architecture_map.md`、`glossary.md`
2. 主线层：`00` 到 `09` 的章节、mini unit 和 `answers.md`
3. 自测层：`final_self_check.md` 和 `run_all_mini_units.py`

当前这套资料的主要优点是：

- 已经有明确的章节主线，不需要新人自己发明阅读顺序。
- 大多数章节都能把“概念验证”和“源码锚点验证”结合起来。
- 业务章节已经明确标注“哪些是已实现平台能力，哪些要结合源码进一步确认”。

本轮重点修补的问题包括：

- 学习入口和阶段计划缺少“怎么学完一章”的闭环说明。
- 个别章节缺少“结合项目分析”，新手容易只记结论，不知道它在真实代码里落在哪里。
- 第 02 章示例里还残留旧模型 ID，和当前项目代码不一致。
- 第 06 章练习说明和 mini unit 的真实函数名不完全一致，容易误导初学者。
- 第 07 章没有点出“只上传文件、不输入文字时会发送默认提示词”这个真实前端行为。

## 修补原则

1. 不重写整套资料，保留原目录和主线。
2. 每章补齐前置知识、本章主线、结合项目分析或源码锚点、常见误区。
3. 最小练习同时保留概念验证和源码锚点验证。
4. 无法从源码可靠确认的内容，明确写“需要结合源码进一步确认”。
5. 不写部署、安装、环境配置，只讲架构设计、代码功能、业务逻辑、模块关系和调用流程。

## 当前剩余注意事项

- 业务知识包中的未来设计边界仍可能继续演进，尤其是第 09 章提到的 PDF ingest/compare 生产模块；这类内容仍要坚持“先 `rg` 真实源码，再更新学习资料”。
- 某些前后端合同可能先在 schema 中出现，但前端还没有完整暴露入口；这种情况要在文档里明确写“需要结合源码进一步确认”，不要替读者脑补。

## 后续维护检查

每次修改 `Study/` 后至少运行：

```bash
python Study/run_all_mini_units.py
git diff --check
```

如果修改了产品行为，则不能只更新 `Study/`，还要按仓库规范补对应测试、E2E 和知识包。
