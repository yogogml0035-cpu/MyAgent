# Study 资料审计记录

本文件记录这套学习资料的维护原则，帮助后续修改时保持“可学习、可验证、可自测”。

## 审计结论

原始资料的章节顺序基本正确：先总览，再后端 API、Storage、Runner、文件资源、流式前端、前端工作区、安全记忆、业务工作流。

主要问题是：

- 多数章节有结论，但缺少“为什么从这里读、下一步读哪里”的推导。
- 部分练习只验证抽象示意代码，没有反查当前源码。
- 第 06 章把展示排序职责说得过于笼统，容易让新人误以为 `mergeExecutionLogs` 负责排序；当前源码中展示排序在 `workspace-view.ts`。
- 第 09 章需要明确“招投标 PDF compare 是知识包业务边界，当前源码主要实现通用 Agent 平台”，避免把规划当成已落地生产模块。

## 修补原则

1. 不重写整套资料，保留原目录和主线。
2. 每章补齐前置知识、本章主线、结合项目分析或源码锚点、常见误区。
3. 最小练习同时保留概念验证和源码锚点验证。
4. 无法从源码可靠确认的内容，明确写“需要结合源码进一步确认”。
5. 不写部署、安装、环境配置，只讲架构设计、代码功能、业务逻辑、模块关系和调用流程。

## 后续维护检查

每次修改 `Study/` 后至少运行：

```bash
python3 Study/run_all_mini_units.py
git diff --check
```

如果修改了产品行为，则不能只更新 `Study/`，还要按仓库规范补对应测试、E2E 和知识包。
