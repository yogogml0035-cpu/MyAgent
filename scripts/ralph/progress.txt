## Codebase Patterns
- Browser-safe project skill discovery lives behind `backend/app/skills/project.py` and locates `backend/skills` via `Path(__file__).resolve().parents[2] / "skills"`, not `settings.skills_dirs`, `MYAGENT_SKILLS_DIRS`, or the current working directory.
- FastAPI feature routes use `APIRouter(prefix="/api", tags=[...])`, response models from `app.schemas`, and explicit registration in `backend/app/main.py`.
- API tests can avoid external Postgres startup by creating `Settings(...)` with `InMemoryTaskStorage(settings.task_root)`.
- LangChain DeepSeek model tests should treat captured API keys as `SecretStr` and compare with `get_secret_value()`.
- Selected skill references are formatted at the backend task message boundary, then the same effective message must be passed to storage, auto-title generation, and `runner.start_background()`.
- Frontend skill option normalization and slash-token helpers live in `frontend/app/skill-selection.ts`; `frontend/app/task-state.ts` re-exports them so API adapters keep using app-level payload helpers.
- Workspace-owned composer state lives in `frontend/hooks/use-task-workspace.ts`; ChatComposer receives explicit props and sends selected skill names through `postTaskMessage`.
- Skill options load as a non-blocking workspace side effect through `fetchSkillOptions()`, with failures surfaced as warning notices so normal text messages remain usable.
- ChatComposer slash UI should stay textarea-based: use `findActiveSkillSlashToken()` / `replaceActiveSkillSlashToken()` for trigger handling, render selected skills as separate chip controls, and let Enter submit only when the skill picker is closed.
- Skill selector styling should stay on dedicated `.skillPicker*` / `.skillChip*` classes with CSS regression assertions in `frontend/tests/workspace/test_workspace_view.test.ts`; do not couple it to `.modelPicker*`.
- Custom local Next dist directories such as `.next-dev-*` must be ignored by both git and ESLint so `npm run lint` only scans source files.
- Full skill selector send-loop E2E should use real frontend/backend HTTP services, inspect the actual browser `POST /api/tasks/{id}/messages` payload for `skills`, then verify persisted user-visible message content through `GET /api/tasks/{id}` and history reload.
