## Codebase Patterns

- 平台级稳定运行边界最好集中沉淀到 `asset/deepagents_platform_knowledge_pack.md` 的单独规则块，而不是只散落在各次 story 的增量条目里；后续核对互斥、reasoning 回传和 run 审计边界时会更直接。
- `TaskConversation` 这类批量 disclosure 交互要以稳定的 post-click `<details open>` 结果为契约；Playwright 不要断言 `requestAnimationFrame` 之间的瞬时状态，否则浏览器调度差异会把正确实现误判成回归。
- `frontend/next.config.mjs` 现在明确把 `next dev` 的输出目录隔离到 `.next-dev`，而生产 `npm run build` 继续写默认 `.next`；如果需要在复用中的 `3001` 开发服务旁边跑 build/typegen/E2E，优先延续这种“dev/build 产物分仓”模式，不要再依赖临时副本。
- 依赖真实 Postgres/Qdrant 合约的 integration test 不应该再直连外部 embedding provider；优先在测试里用本地 fake embedding 保持向量维度与 recall/upsert 语义，避免 DashScope/SSL 抖动把无关回归拖挂。
- 后端运行状态以 `task_id`/`run_id` 为边界，`storage.start_run()`、`TaskRunner.start_background()`、`append_event()` 和终端事件需要保持同一个 `run_id`。
- `TaskRunner` 的流式事件最好在两个边界都做 run 归属收敛：`start()` 在 `convert_stream_event()` 之后立刻规范化 `EventRecord.run_id`，`start_background()` 在持久化前再次规范化，这样即使上游 callback 或 fake runner 带错 `run_id`，事件也不会串到别的 run。
- 前端字段转换集中在 `frontend/app/task-state.ts`，默认渲染应使用简洁 live metadata，完整运行证据通过 `rawRecord`、expanded diagnostics 和 JSONL copy 暴露。
- `reasoning_content` 是 run 级诊断证据，只能保存在当前 run 的事件日志与展开诊断中，不能进入普通 messages、conversation context、长期记忆、Qdrant payload 或默认 artifact。
- `ConversationContextBuilder` 和 long-term memory 只能消费 canonical 的 `TaskState.messages`、当前用户目标和最终答案；如果候选文本带 `reasoning_content`、`assistant_thinking_delta`、`assistant_answer_delta`、`tool_call_id` 或 `tool_result` 这类 run 诊断标记，必须在 context/memory 边界直接拒收。
- 行为变更必须同步测试、实际浏览器 E2E、截图证据和 `asset/deepagents_platform_knowledge_pack.md`。
- OpenAI-compatible provider payload 表征测试可以先用 `langchain_openai.chat_models.base._convert_message_to_dict()` 复用 LangChain 既有序列化，再由本地 adapter 精确补回 provider 专属字段。
- DeepSeek message adapter 需要和 `ChatDeepSeek._get_request_payload()` 保持一致：assistant content list 要折叠为字符串、tool content list 要 JSON 化，再只对当前 assistant tool-call turn 回放 `reasoning_content`。
- `app/models/provider.py#create_model()` 是切换 provider wrapper 的稳定入口：non-thinking ID 保持 `ChatDeepSeek`，thinking-enabled ID 才切到 `DeepSeekThinkingChatModel`，这样上层调用方不用感知 adapter。
- `backend/app/streaming/v2_adapter.py` 的 `thinking_chunk` 既是完整 reasoning 载体，也是 tool-call 诊断挂点：如果同一 provider turn 带了工具调用，优先把 `tool_call_id`/`tool_call_ids`/`tool_calls` 放进 thinking payload，再由 `event_converter.py` 保持稳定的 `payload.live.display_text`，不要把这些诊断字段塞进 collapsed 文案。
- `backend/app/streaming/event_converter.py` 必须把 `assistant_thinking_delta` 的完整 reasoning 放在 `payload.content`，同时让顶层 `message` 和 `payload.live.display_text` 都保持稳定状态文案；这样即使旧消费方回退到 `message`，也不会把 reasoning 泄露到默认 UI。
- `backend/app/api/tasks.py#send_message()` 的同 task 互斥是双层保护：先检查 `runner.is_running(task_id)` 立即返回 409，再由 `storage.start_run(... expected_statuses=...)` 兜底阻止非允许状态下创建新 run。
- `storage.read_events(task_id, after_id=..., run_id=...)` 的游标语义必须先按 task 级 `seq` 解析 `after_id`，再应用可选 `run_id` 过滤；不要把恢复语义改成 run-local cursor，否则会破坏 SSE/HTTP polling 的 fail-open 恢复。
- `frontend/hooks/use-task-workspace.ts` 的 busy 作用域要拆开：`ChatComposer` 只消费当前会话提交/停止/切换状态，`ChatSidebar` 与 clear-all 单独消费历史 mutation busy；否则一个局部请求会误伤跨会话发送和切换。
- 前端切换会话时，如果 composer/日志需要立刻反映“当前会话”的运行态，先用目标 `taskSummary` 同步 `taskId`/`status` 并清空旧会话内容，再异步 `fetchTask(id)`；否则 A 会话的运行 placeholder 或 stop 控件会在 B 会话加载期间残留。
- `frontend/app/task-state.ts` 必须保留 `assistant_thinking_delta` / `assistant_answer_delta` 的完整 `payload.content` 作为 diagnostics 源数据；任何截断都只能发生在 collapsed preview 投影层，不能提前发生在 state normalization，否则 expanded JSON 和逐行复制都会永久丢内容。
- `frontend/app/workspace-view.ts` 的 thinking/answer diagnostics 聚合必须以当前 run segment 为边界：expanded JSON 用当前 run 的 stream 片段重建，整轮复制始终基于 `RunActivityGroup.logs` 输出 raw JSONL，避免不同 run 的 reasoning/final answer 串线。
- `TaskConversation` 的 run 卡片要把“默认简洁进度”和“完整 run 审计”分开：collapsed rows 继续只显示稳定状态文案，整轮审计则用 `buildRunDiagnosticsJson(group.logs)` 渲染 pretty JSON；`buildLogClipboardText(group.logs)` 继续输出同一 run 的 JSONL，并在 raw record 缺字段时补齐 `run_id`、`created_at`、`seq` 等稳定元数据。
- `backend/app/storage.py#list_task_summaries()` 只会把“至少有一条非空 user message”的 task 暴露给前端历史栏；浏览器 E2E 如果想让合成会话出现在侧栏，不能只改 `tasks.title`，还要补一条 user message。
- `frontend/e2e-playwright` 的稳定验收模式是“真实 3001/8001 服务 + 公共 API 建 task + 直接向 Postgres seed 受控 run/events”；这样既保留真实浏览器/API 合约，又不会把 E2E 稳定性绑在外部 provider 上。
