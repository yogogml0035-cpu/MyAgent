# Requirements: 任务日志与文件产物体验优化

## 来源

- 生成自当前对话中的需求对齐内容。
- 关联 PRD：`prd.md`

## 原始对齐需求

当前项目需要优化长任务运行体验和文件交付体验。用户在 DeepSeek thinking 模式下运行任务时，进度日志会产生 2000+ 条事件；一旦点击日志内容或滚动日志区域，前端出现浏览器 Page Unresponsive 弹窗，导致无法查看结果，也无法点击“暂停生成”。用户只能强制关闭终端。主对话页面不能为了展示调试日志而牺牲滚动、暂停生成、查看结果这些核心操作。

用户也遇到文件交付问题：非 thinking 模式下最终回复生成出来了，但没有把用户要求的 Word 文件作为可下载产物提供。AI 只在回复中写出类似 `/root/技术参数总结.docx` 的路径，用户无法点击、无法下载，也无法确认真实文件是否已生成。

## 范围

### 包含

- 主对话默认只显示任务阶段、当前动作、错误/警告、最终完成状态和产物入口。
- 2000+ 原始事件、完整 reasoning/tool JSON 不在主页面实时渲染。
- 完整日志提供下载入口，用于诊断和审计。
- 用户明确要求的交付文件必须作为可下载产物展示。
- AI 在最终回复里提到“已保存/已生成”的文件必须被登记为产物并展示下载入口。
- 最终回复使用“简短摘要 + 文件下载卡片”，不把完整正文全部塞进聊天气泡。
- 如果用户要求生成文件但没有可下载产物，任务不能算成功，必须提示文件未生成或未登记为产物，并提供重新生成或修复入口。

### 不包含

- 临时文件、内部 JSON、草稿、工具中间文件的默认展示。
- 把完整原始日志继续作为默认主页面内容。
- 暴露容器路径、本地私有路径、provider 密钥、数据库 URL、Qdrant URL 或客户资料。

## 业务场景

### 场景 1：长任务运行中的进度日志

- 用户发送任务后，主对话只显示关键阶段和当前状态。
- thinking 模式产生大量事件时，页面仍然可以滚动。
- 日志区点击、滚动、折叠/展开不能触发浏览器无响应。
- 用户可以在任务运行中点击暂停生成。
- 完整日志作为诊断文件下载，不在主对话中实时渲染完整 JSON。

### 场景 2：用户要求生成 Word/PPT/Excel/报告文件

- 只要用户明确要求交付文件，最终结果必须提供可点击下载卡片。
- 最终聊天气泡只说明已完成、概括核心内容、列出文件。
- 文件卡片是正式交付入口。

### 场景 3：AI 声称已生成文件

- 如果最终回复出现“已保存”“已生成”“Word 文档已保存到...”等表达，对应文件必须作为产物展示。
- 不允许只显示 `/root/技术参数总结.docx` 这类用户不可点击、不可下载的路径。

### 场景 4：交付文件缺失

- 用户明确要求生成文件，但系统没有可下载产物时，任务状态不能显示为成功。
- UI 必须明确提示“文件未生成或未登记为产物”。
- 用户应能看到重新生成或修复的入口，而不是虚假的“文档已生成完毕”。

## 验收口径

- thinking 模式产生大量事件时，主页面仍可正常滚动，点击日志区不会卡死，暂停生成可用。
- 进度日志默认只展示关键阶段；完整诊断通过下载获得。
- 生成 Word 等文件后，最终回复旁边能看到文件下载卡片。
- 最终回复是“简短摘要 + 文件入口”，不是完整正文替代文件。
- 文件缺失时，用户看到明确失败/修正提示，而不是虚假的完成状态。
