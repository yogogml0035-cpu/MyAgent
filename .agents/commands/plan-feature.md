# 规划一个新任务

## 功能: $ARGUMENTS

## 使命

通过系统化的代码库分析、外部调研和策略规划，把一个功能需求转化为**完整的实现计划**。

**核心原则**：这个阶段不写代码。目标是产出一份上下文充分、可直接执行的实现计划，让 AI 代理能够一次性高成功率落地。

**关键理念**：上下文为王。计划必须包含实现所需的全部信息，包括模式、必读文件、文档、验证命令等，让执行代理第一次尝试就能成功。

## 规划流程

### 阶段 1：理解功能

**深入分析功能：**

- 提炼要解决的核心问题
- 识别用户价值与业务影响
- 判断功能类型：新能力 / 增强 / 重构 / 缺陷修复
- 评估复杂度：低 / 中 / 高
- 映射会受影响的系统与组件

**创建用户故事格式，或在用户已提供用户故事时进行完善：**

```
作为一名<用户类型>
我想要<动作/目标>
以便<收益/价值>
```

### 阶段 2：收集代码库情报

**使用专门代理并进行并行分析：**

**1. 项目结构分析**

- 识别主要语言、框架与运行时版本
- 梳理目录结构与架构模式
- 确认服务 / 组件边界与集成点
- 定位配置文件（如 `pyproject.toml`、`package.json` 等）
- 找出环境初始化与构建流程

**2. 模式识别**（适合时使用专门子代理）

- 在代码库中搜索相似实现
- 识别编码约定：
  - 命名模式（CamelCase、snake_case、kebab-case）
  - 文件组织与模块结构
  - 错误处理方式
  - 日志模式与规范
- 提炼该功能所属领域的通用模式
- 记录需要避免的反模式
- 检查 `CLAUDE.md` 中的项目专属规则与约定

**3. 依赖分析**

- 梳理与该功能相关的外部库
- 理解这些库是如何被集成的（查看 imports、配置等）
- 查找 `docs/`、`ai_docs/`、`.agents/reference` 或 `ai-wiki` 中的相关文档（如果存在）
- 记录库版本与兼容性要求

**4. 测试模式**

- 识别测试框架与测试结构（如 pytest、jest 等）
- 找到类似测试作为参考
- 理解测试组织方式（单元 / 集成）
- 记录覆盖率要求与测试规范

**5. 集成点**

- 识别需要更新的现有文件
- 确定需要创建的新文件及其位置
- 梳理路由 / API 注册模式
- 如适用，理解数据库 / 模型模式
- 如相关，识别认证 / 授权模式

**澄清歧义：**

- 如果在这一阶段需求仍不清晰，先向用户确认，再继续
- 获取具体实现偏好（库、方案、模式）
- 在继续之前先解决架构决策分歧

### 阶段 3：外部调研与文档收集

**在适合时使用专门子代理做外部调研：**

**收集文档：**

- 调研最新库版本与最佳实践
- 查找带具体锚点的官方文档
- 搜集实现示例与教程
- 识别常见坑点与已知问题
- 检查破坏性变更与迁移指南

**技术趋势：**

- 调研当前技术栈的最佳实践
- 查找相关博客、指南或案例研究
- 识别性能优化模式
- 记录安全注意事项

**整理调研引用：**

```markdown
## 相关文档

- [库官方文档](https://example.com/docs#section)
  - 具体功能实现指南
  - 原因：实现 X 功能所必需
- [框架指南](https://example.com/guide#integration)
  - 集成模式章节
  - 原因：说明如何连接组件
```

### 阶段 4：深度策略思考

**重点深入思考：**

- 这个功能如何融入现有架构？
- 关键依赖与实施顺序是什么？
- 可能出什么问题？（边界情况、竞态、错误）
- 如何进行完整测试？
- 有哪些性能影响？
- 是否存在安全考虑？
- 这种方案的可维护性如何？

**设计决策：**

- 在备选方案中进行选择，并给出清晰理由
- 为可扩展性与未来修改做设计
- 如有需要，为向后兼容做规划
- 考虑可伸缩性影响
