# AGENTS 设计原则

## 目标

`AGENTS.md` 是给代理看的导航与执行入口，不是完整系统手册。

## 分层

- 根 `AGENTS.md`：全局索引层，描述入口、阅读顺序、维护规则
- 根 `ARCHITECTURE.md`：系统架构层，描述系统边界、子系统职责、理解路径
- 系统级补充文档：集成说明层，描述接口、协作、跨项目关系
- `.planning/codebase/`：子系统事实层，描述代码现状
- 更深层级的 `AGENTS.md`：局部规则层，只约束其作用域内目录

## 生成顺序

生成或刷新时，优先遵循：

1. 读取子项目 `.planning/codebase/` 事实
2. 提炼并刷新根级 `ARCHITECTURE.md`
3. 提炼并刷新系统级补充文档
4. 再生成根级 `AGENTS.md` 作为全局索引

也就是说：

- 事实先沉淀在子项目
- 系统总览先沉淀在 `ARCHITECTURE.md`
- 关系先沉淀在系统级补充文档
- 入口最后沉淀在 `AGENTS.md`

## 根 AGENTS.md 应强调

- 子系统边界
- 文档分层
- 阅读顺序
- 维护责任
- 稳定入口

## 根 AGENTS.md 应避免

- 实现细节
- 架构摘要再压缩
- 高频变化的文件级说明
- 与 `.planning/codebase/` 重复的内容

## 一条判断启发式

如果一句话回答的是“代理应该先看哪里、边界怎么判断、文档怎么分层”，它更适合进入 `AGENTS.md`。

如果一句话回答的是“这个仓库整体是怎么划分的、系统边界在哪里、应该怎么理解整套系统”，它更适合进入 `ARCHITECTURE.md`。

如果一句话回答的是“多个子系统之间怎么协作、怎么划分、怎么集成”，它更适合进入系统级补充文档。

如果一句话回答的是“这个子系统内部是怎么实现的”，它更适合留在子项目自己的 `.planning/codebase/`。
