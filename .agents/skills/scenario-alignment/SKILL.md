# Scenario Alignment

把用户输入的大段需求、bug、想法或计划整理成可快速审核的业务 PRD 对齐稿。此 skill 只关注“用户真实想要什么、场景是否完整、产品口径是否一致”，不要写技术方案，不要拆实施任务，不要改代码。

核心目标：**用极精炼的文字达成共同理解，并形成后续技术合同 / 实施设计可消费的业务需求文档。**

“访谈式对齐”不是输出很多问题，而是沿着会影响需求方向的决策树逐个排除歧义。能从代码、harness 或用户材料中查到的问题，直接查证，不要问用户。

## 输入

- 用户的一段或多段功能想法、目标、bug、现象、抱怨或口述需求
- 可选：相关页面、接口、旧 PRD、截图、业务材料路径
- 当前仓库的 harness  / 架构文档 / 相关代码事实

## 工作流程

1. 读取项目约束：
   - 先读根级 `AGENTS.md`
   - 按 `AGENTS.md` 的阅读顺序读取相关 architecture / codebase 文档
   - 只进入与本功能有关的代码、页面、接口和数据结构
2. 从用户输入中抽取业务意图，不复述原文：
   - 用户要达成的目标
   - 涉及的角色、入口、对象和状态
   - 明确提出的需求、bug、禁用项、命名、按钮、权限、可见性、生成 / 审核 / 保存 / 下载规则
   - 待确认的产品判断
3. 快速查证相关代码事实，只保留会影响产品口径的事实：
   - 已存在的入口、页面、接口、服务或状态
   - 当前行为与目标行为的关键差异
   - 不能破坏的历史语义
4. 沿决策分支对齐需求：
   - 先解决会阻塞后续场景定义的上游决策
   - 再整理成功路径、失败 / 禁止路径、权限与可见性
   - 最后确认历史数据、默认值、边界状态和例外处理
5. 如仍有关键业务不确定点，最多提出 1-3 个问题。每个问题必须附推荐答案；问题必须能改变产品方向或验收口径。
6. 输出一份精炼业务 PRD 对齐稿；如果问题会阻塞文档成立，则先只问问题，不强行生成。

## 提问规则

- 问题要少而关键，最多 3 个；如果超过 3 个，先问最上游的 1-3 个。
- 每个问题都必须说明推荐答案，格式为“推荐：...”。
- 不问能通过代码、文档、截图或用户已给材料判断的问题。
- 不问实现方式问题，例如数据库字段、接口设计、组件拆分、任务顺序。
- 不用“是否需要考虑...”这类泛问题；改成具体决策，例如“历史记录是否继续对原创建人可见？推荐：继续可见，避免破坏审计链路。”

## 输出格式

```markdown
# [功能名] 业务 PRD 对齐稿

## 目标
- [一句话说明用户要达成的业务结果]

## 范围
- 包含：[本次明确要做的内容]
- 不包含：[用户明确排除或本阶段不处理的内容；没有则写“无”]

## 业务场景
场景1：[场景标题]
- [角色 / 入口 / 触发动作]
- [目标行为、当前问题或要调整的内容]
- [重要边界：权限、可见性、历史记录、生成 / 保存 / 下载规则等]

场景2：[场景标题]
- ...

## 验收口径
- [用户可观察到的业务结果，不写技术实现]

## 需要对齐的问题
1. ...
   推荐：...
```

## 输出压缩规则

- 优先保证完整，不因为篇幅限制漏掉用户明确提出的需求、bug、命名调整、按钮去留、登录注册变化、角色权限、数据可见性、生成/审核/保存/下载规则。
- 文档必须精炼；每个小节只写能帮助用户确认产品口径的内容。
- 场景数量按需求自然拆分；简单需求可以 1-3 个场景，复杂需求可以超过 5 个场景，但不要为了形式拆场景。
- 每个场景可以多行，但只写业务口径，不写数据库设计、接口字段、组件拆分或实施步骤。
- 验收口径只写用户可感知的结果；不要写 Given / When / Then，不写测试用例矩阵。
- 对用户明确点名的页面、入口、角色、按钮、文案、中英文名称，应尽量保留在对应场景里。
- 只有会改变产品方向或验收口径的问题才放到 `需要对齐的问题`，最多 3 个；没有就写“无”。
- 不单独输出“当前代码事实”，除非某个事实会改变场景判断；需要时合并进对应场景或确认问题。
- 不复述用户输入，不写背景铺垫，不写“我将会”。
- 如果需求和 bug 混在一起，按实际业务场景归并；必要时在场景标题中标注“问题修正”或“新需求”。

## 禁止内容

- 不要写数据库设计、接口字段、组件拆分、实施顺序。
- 不要写技术 PRD、实施设计、user stories、开发任务、接口合同。
- 不要输出大段代码事实清单。
- 不要把能从代码中查到的问题丢给用户。
- 如果用户要求保存文件，默认保存到 `docs/需求文档/<feature-name>/01-scenario-alignment.md`。
- 如果用户要求继续下一步，明确建议使用 `technical-contract-review`。
