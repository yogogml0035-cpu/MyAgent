# 创建或优化全局规则

通过分析代码库并提取模式，创建或维护项目根目录的 `AGENTS.md` 文件。

---

## 目标

创建或优化适用于当前项目的全局规则，为 AI 编码代理提供以下上下文信息：
- 这个项目是什么
- 使用了哪些技术
- 代码是如何组织的
- 需要遵循哪些模式和约定
- 如何构建、测试与验证

---

## 阶段 0：选择模式

先在当前项目根目录检查是否存在 `AGENTS.md`：

- **不存在 `AGENTS.md`**：进入创建模式，使用下方“内置 AGENTS.md 模板”初始化文件。
- **已经存在 `AGENTS.md`**：进入优化模式，读取现有文件，并按照模板结构整理、补充和完善。

### 优化模式原则

优化已有 `AGENTS.md` 时：

- 保留已有的项目特有规则、人工写入的约束、关键命令和注意事项。
- 按内置模板的章节顺序重新组织内容，补齐缺失但有价值的章节。
- 删除模板占位符、空章节、重复说明和与当前代码库明显不符的内容。
- 如果现有说明与代码库证据冲突，以当前代码库为准；冲突不确定时，在输出中标记需要用户确认。
- 只写对编码代理有帮助的信息，避免把 `AGENTS.md` 变成完整项目文档。

## 阶段 1：发现

### 识别项目类型

首先，判断这是什么类型的项目：

| 类型 | 判断依据 |
|------|----------|
| Web 应用（全栈） | 有独立的 client/server 目录，或存在 API 路由 |
| Web 应用（前端） | 使用 React/Vue/Svelte，且没有服务端代码 |
| API/后端 | 使用后端/API 框架或路由层，且没有前端 |
| 库/包 | `package.json` 中有 `main`/`exports`，可发布 |
| CLI 工具 | `package.json` 中有 `bin`，提供命令行接口 |
| Monorepo | 有多个 package，或使用 workspaces 配置 |
| 脚本/自动化 | 独立脚本为主，面向任务执行 |
| Python 项目 | 存在 `pyproject.toml`、`requirements.txt`、`setup.py` 或 Python 包目录 |
| Go/Rust/Java 项目 | 存在 `go.mod`、`Cargo.toml`、`pom.xml`、`build.gradle` 等 |
| 基础设施/运维 | 以 Docker、Terraform、Kubernetes、CI/CD 配置为主 |

### 分析配置

查看根目录中的配置文件：

```
package.json / pnpm-workspace.yaml  → JS/TS 依赖、脚本、工作区
pyproject.toml / requirements.txt   → Python 依赖、工具、打包配置
go.mod / Cargo.toml / pom.xml       → Go/Rust/Java 项目配置
tsconfig.json / vite.config.*       → TypeScript 与前端构建配置
Dockerfile / docker-compose.yml     → 容器与本地服务
Makefile / justfile / Taskfile.yml  → 常用任务命令
.github/workflows / .gitlab-ci.yml  → CI 验证流程
*.config.*                          → 各类工具配置
```

### 梳理目录结构

浏览代码库，理解其组织方式：
- 源代码放在哪里？
- 测试文件在哪里？
- 是否有共享代码？
- 配置文件位于哪里？

---

## 阶段 2：分析

### 提取技术栈

从项目配置文件中识别：
- 运行时/语言（Node、Bun、Deno、Python、Go、Rust、Java、浏览器等）
- 框架
- 数据库（如果有）
- 测试工具
- 构建工具
- Lint/格式化工具

### 识别模式

研究现有代码中的以下模式：
- **命名**：文件、函数、类是如何命名的？
- **结构**：文件内部是如何组织代码的？
- **错误处理**：错误是如何创建和处理的？
- **类型**：类型/接口是如何定义的？
- **测试**：测试是如何组织的？

### 找出关键文件

识别理解项目时必须关注的重要文件：
- 入口文件
- 配置文件
- 核心业务逻辑
- 共享工具
- 类型定义

---

## 阶段 3：生成或优化

### 写入 AGENTS.md

使用下方“内置 AGENTS.md 模板”作为创建和优化的统一结构，不需要再读取外部模板文件。

**输出路径**：`AGENTS.md`（项目根目录）

**创建模式：**
- 将内置模板复制为新的 `AGENTS.md`
- 根据发现与分析阶段的信息替换占位内容
- 删除不适用的章节和所有模板注释

**优化模式：**
- 将已有内容映射到模板章节中
- 保留有用内容并改写为更清晰、可扫描的规则
- 补充缺失的技术栈、命令、结构、测试、验证和关键文件信息
- 删除过时、重复或无法从代码库验证的泛泛描述

**按项目实际情况调整：**
- 删除不适用的章节
- 添加适合当前项目类型的章节
- 保持简洁，只保留真正有用的信息

**建议包含的核心章节：**

1. **项目概览** - 这是什么项目？它的作用是什么？
2. **技术栈** - 使用了哪些技术？
3. **命令** - 如何开发、构建、测试、lint？
4. **结构** - 代码是如何组织的？
5. **模式** - 应遵循哪些约定？
6. **关键文件** - 哪些文件值得重点了解？

**可选章节（如适用可添加）：**
- 架构（适用于复杂应用）
- API 端点（适用于后端）
- 组件模式（适用于前端）
- 数据库模式（如果使用数据库）
- 按需上下文参考

## 提示

- 让 `AGENTS.md` 保持聚焦、易于快速浏览
- 不要重复其他文档中已有的信息，优先使用链接
- 重点写模式和约定，而不是面面俱到的说明文档
- 随着项目演进持续更新它
